# 校园二手交易平台 - 对话记录汇总

## 对话一：项目部署与初始上线

用户将校园二手交易平台项目部署到阿里云服务器（47.96.152.231），过程中遇到并解决了以下问题：

1. **端口安全组未放行** — 在阿里云控制台安全组中添加入方向规则，放行 TCP 8080 端口
2. **JAR 包不完整** — 服务器上 jar 只有 277KB，缺少 Spring Boot 启动入口，报 `no main manifest attribute`。原因是本地 Java 进程锁定了 target 目录下的 jar 文件，导致 Maven 无法完整打包。杀掉占用进程后重新编译，生成 33MB 完整 jar 包
3. **API 路径不匹配** — 后端 Controller 映射的是 `/api/product`（单数），测试时用了 `/api/products`（复数），导致 500 错误。使用正确路径后正常
4. **图片无法显示** — 服务器上传目录为空，图片文件未同步，且配置文件缺少 `campustrade.upload.path` 配置项。通过 SFTP 上传图片并补充配置解决
5. **配置文件项名不一致** — FileController 使用 `@Value("${campustrade.upload.path}")`，但服务器 application.yml 中只有 `upload.path`。补充完整配置项后解决

部署完成后服务运行正常，已部署 10 件示例商品，数据库连接、图片访问、所有 API 接口均正常响应。

## 对话二：下架商品无法重新上架 & 测试账户无法发布视频

### 问题 1：下架商品无法重新上架

**原因分析**：前端 MyProducts.vue 中"重新上架"按钮调用 `api.updateProduct({ id, status: 1 })`，但后端 ProductController.update() 方法完全忽略了 status 字段。ProductSaveDTO 中没有 status 字段，update 方法里也没有 setStatus() 调用，导致前端传的 status:1 被丢弃，数据库中 status 保持为 3（已下架）。

**修复方案**：
- 后端新增 `PUT /api/product/relist` 专用接口，通过商品 ID 将 status 从 3 改回 1
- ProductSaveDTO 中添加 `status` 字段（Integer 类型，1在售/2已售/3已下架）
- ProductController.update() 方法中增加对 status 字段的处理：`if (dto.getStatus() != null) product.setStatus(dto.getStatus())`
- 前端 api/index.js 新增 `relistProduct` 方法，MyProducts.vue 改为调用 `api.relistProduct(id)`

### 问题 2：测试账户无法发布视频

**原因分析**：服务器上之前部署的 jar 包中，ProductController.publish() 方法没有 setVideoUrl() 调用。原因是 replace_in_file 工具在本地多次静默失败（工具返回成功但文件实际未被修改），导致代码没有真正写入文件。

**修复方案**：
- 确认本地源代码已正确包含 `setVideoUrl(dto.getVideoUrl())` 调用
- 重新编译后通过 javap 验证 class 文件包含 `setVideoUrl` 调用
- 构建前端（npm run build）后，将新 jar 包和前端 dist 目录通过 SFTP 上传到服务器
- 重启服务（pkill 旧进程 + nohup 启动新 jar）
- 验证 relist 接口和首页均正常响应（HTTP 200）

### 部署后验证

- 服务端口 8080 正常监听
- 首页访问返回 200
- relist 接口正常响应（返回 JSON 格式错误信息而非 404，说明接口已正确注册）

## 项目技术信息汇总

- **后端**：Spring Boot 3.2.5 + MyBatis-Plus 3.5.6 + MySQL 8.0
- **前端**：Vue 3 + Vite
- **Java 版本**：OpenJDK (Zulu) 22.0.2
- **服务器**：阿里云 ECS (CentOS)，公网 IP 47.96.152.231
- **部署路径**：`/opt/campus-trade/`

## 维护经验总结

1. **工具可靠性**：write_to_file 和 replace_in_file 工具在某些情况下会"静默成功"（返回成功但文件实际未修改）。写文件后必须用 read_file 验证，必要时使用 PowerShell 或 Python 脚本直接写入
2. **编译前检查**：编译前必须确保没有 Java 进程占用 jar 包（pkill 或 taskkill），否则 Maven 无法完整打包
3. **部署验证**：每次部署后必须通过 curl 测试关键接口，确认服务正常启动后再告知用户
4. **代码修改验证**：修改 Java 代码后建议用 javap 反编译 class 文件确认修改已生效
