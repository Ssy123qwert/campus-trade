## 二、经典问题详解（共 7 个）

---

### 问题 1：阿里云安全组未放行 8080 端口

**现象**：外部浏览器无法访问 `http://47.96.152.231:8080`，但服务器内部 `curl localhost:8080` 正常。

**原因**：阿里云 ECS 默认安全组只开放了 22 (SSH) 等常用端口，8080 未在入方向规则中。

**解决**：登录阿里云控制台 → 安全组 → 添加入方向规则 → 协议 TCP、端口 8080、授权对象 0.0.0.0/0。

**经验**：部署 Web 服务第一步永远是检查安全组/防火墙。可通过 `ss -tlnp | grep 8080` 确认服务在服务器内监听正常，再通过外部 `curl` 或浏览器测试，失败则大概率是网络层问题。

---

### 问题 2：JAR 包不完整——只有 277KB 而非预期的 33MB

**现象**：服务器启动 jar 时报错 `no main manifest attribute, in campus-trade-server-1.0.0.jar`。

**诊断过程**：
- 查看服务器上 jar 文件大小：`ls -la /opt/campus-trade/` → 只有 282720 字节（约 277KB）
- 正确的 Spring Boot fat jar 应该在 33MB 左右
- 这说明 Maven 打包不完整，jar 里面没有 Spring Boot 的启动类

**原因**：本地有一个 Java 进程（PID 6168 和 21056）持有了 target 目录下 jar 文件的锁，导致 Maven 的 `spring-boot-maven-plugin` 无法正常将依赖打包进去，生成的是一个残缺 jar。

**解决**：
1. 先在本地用 `taskkill /F /PID 6168` 和 `taskkill /F /PID 21056` 杀掉占用进程
2. 删除 target 目录：`rmdir /s /q target`
3. 重新编译：`mvn clean package -DskipTests`
4. 验证 jar 大小：确认 target 下的 jar 约 33MB
5. 通过 SFTP 上传到服务器

**经验**：编译前务必确保没有进程占用输出文件。Maven 打包失败不会报错，只是静默产出残缺 jar。可以用 `jar tf xxx.jar | findstr "Main-Class"` 或者直接看文件大小来判断完整性。

---

### 问题 3：API 路径单复数不匹配

**现象**：访问 `/api/products` 返回 500 错误，Spring Boot 日志显示 `No static resource api/products`。

**原因**：后端 Controller 上标注的映射是 `@RequestMapping("/api/product")`（单数），但测试时用了 `/api/products`（复数）。Spring Boot 找不到匹配的 Controller，尝试作为静态资源处理，返回 500。

**解决**：统一使用 `/api/product/list`、`/api/product/detail?id=xx` 等正确路径。

**经验**：
- 项目初期就统一 API 命名规范（单数/复数、驼峰/下划线）
- 500 错误如果日志里有 "No static resource" 字样，优先检查 URL 拼写

---

### 问题 4：图片无法显示（两个子问题）

**子问题 A - 服务器上传目录为空**

**现象**：前端能正常展示商品标题、价格等文字信息，但所有图片都是空白/裂图。

**原因**：本地开发时图片保存在 `e:/CodeBuddy-work space/campus-trade/campus-trade-server/uploads/images/` 目录，但部署到服务器后，`/opt/campus-trade/uploads/images/` 目录是空的——图片文件没有随项目一起上传。

**解决**：通过 SFTP 将本地 6 张示例商品图片上传到服务器对应目录。

**子问题 B - 配置项名称不一致**

**现象**：图片上传成功但返回的 URL 无法访问。

**原因**：`FileController.java` 中使用 `@Value("${campustrade.upload.path}")` 读取上传路径，但服务器 `application.yml` 中配置的是：
```yaml
upload:
  path: /opt/campus-trade/uploads
```
应改为：
```yaml
campustrade:
  upload:
    path: /opt/campus-trade/uploads
```

**解决**：修正服务器端 application.yml 中的配置项名称，使其与 @Value 注解中的变量名完全一致。

**经验**：
- 部署时记得同步静态资源文件（图片、模板等）
- 配置文件项名称必须与 `@Value` 注解完全匹配，Spring 不会自动做模糊匹配
- 本地和服务器使用同一份 application.yml 模板可以减少此类问题

### 问题 5：下架商品无法重新上架（前后端联调缺陷）

**现象**：用户在"我的商品"页面点击"重新上架"按钮，提示操作成功但商品状态没有变化，仍在"已下架"状态。

**根因分析**：
1. 前端 `MyProducts.vue` 中"重新上架"按钮调用了 `api.updateProduct({ id, status: 1 })`
2. 后端 `ProductController.update()` 方法的 `ProductSaveDTO` 参数中根本没有 `status` 字段
3. `update()` 方法里也没有 `product.setStatus()` 的逻辑
4. 结果前端传的 `status: 1` 被 Spring 静默丢弃，数据库中 status 始终是 3

**修复方案**：

后端改动：
1. 在 `ProductSaveDTO` 中添加 `status` 字段（Integer 类型，1=在售，2=已售，3=已下架）
2. 在 `ProductController.update()` 方法中添加：`if (dto.getStatus() != null) product.setStatus(dto.getStatus());`
3. 新增专用接口 `PUT /api/product/relist`，通过商品 ID 直接将 status 从 3 改为 1

前端改动：
1. 在 `api/index.js` 中新增 `relistProduct: (id) => request("/product/relist?id=" + id, { method: "PUT" })`
2. 在 `MyProducts.vue` 中将重新上架按钮改为调用 `api.relistProduct(row.id)`

**经验**：
- 前后端数据传输时，DTO 必须包含所有前端可能传递的字段
- 对于状态变更这类操作，建议使用专用接口（如 `/relist`、`/offline`）而非通用 update 接口
- 前端传参前要确认后端 API 是否真的接收并处理了该参数

---

### 问题 6：测试账户无法发布视频

**现象**：测试账户登录后，发布商品时选择了视频文件，但发布成功后数据库中 video_url 字段为空，视频没有保存。

**根因分析**：
1. 查看服务器上部署的 jar 包中 `ProductController.class`，发现 `publish()` 方法中没有 `setVideoUrl()` 调用
2. 本地源代码中明明有 `setVideoUrl()`——那为什么编译出来的 class 里没有？
3. 回溯发现是 CodeBuddy 的 `replace_in_file` 工具存在"静默失败"问题：工具返回"替换成功"，但文件实际未被修改
4. 当时代码只存在于对话中，没有被真正写入文件，导致编译时缺失了视频处理逻辑

**修复方案**：
1. 用 Python/PowerShell 脚本直接写入文件，绕过 replace_in_file 工具
2. 确认本地源代码正确后，用 `javap -c -p ProductController.class` 反编译 class 文件，验证 `setVideoUrl` 调用存在
3. 重新编译：`mvn clean package -DskipTests`
4. 构建前端：`npm run build`
5. 通过 SFTP 上传新 jar 和 dist 目录到服务器
6. SSH 重启服务：`pkill -f campus-trade-server && nohup java -jar ...`

**经验**：
- 不要盲目信任工具返回的"成功"状态，写文件后必须用 read_file 或 javap 等手段验证
- 对于关键修改，备用 PowerShell/Python 直接写入文件
- 部署后务必用 curl 测试关键接口

---

### 问题 7：replace_in_file 工具"静默成功"（工具链问题）

**现象**：`replace_in_file` 工具多次返回 "success"，但用 `read_file` 验证发现文件内容完全没有变化。

**常见原因**：
1. IDE 文件缓存未刷新，工具认为写入成功但实际磁盘未变更
2. 文件被其他进程（如 Vite 热更新、Java 进程）锁定
3. `old_str` 参数中的空格、换行符与实际文件不完全一致，导致匹配失败但被误报为成功

**应对策略**：
1. 每次写文件后立即用 `read_file` 验证
2. 如果连续 3 次写入失败，改用 PowerShell 的 `[System.IO.File]::WriteAllText()` 或 Python 直接写入
3. 需要先用 `taskkill` 或 `pkill` 停掉占用文件的进程
4. 对于 Java 代码，写入后可以用 `javap` 反编译 class 来确认最终效果

## 六、维护备忘

如需修改代码并更新到服务器，标准流程：

1. 本地修改代码
2. **停掉占用 jar 的进程**：`taskkill /F /PID xxx`
3. 编译：`mvn clean package -DskipTests`
4. **验证**：`javap -c -p target/classes/.../xxx.class` 确认修改生效
5. 构建前端：`cd campus-trade-ui && npm run build`
6. SFTP 上传 jar 和 dist 到服务器 `/opt/campus-trade/`
7. SSH 重启：
```bash
pkill -f campus-trade-server
cd /opt/campus-trade
nohup java -jar campus-trade-server-1.0.0.jar \
  --spring.config.location=file:/opt/campus-trade/application.yml \
  > app.log 2>&1 &
```
8. 验证：`curl -s -o /dev/null -w "%{http_code}" http://localhost:8080/` 应返回 200
9. 测试关键接口：`curl -X PUT "http://localhost:8080/api/product/relist?id=xx"` 等

## 七、核心教训

1. 部署时安全组 > 端口监听 > 日志排查，按此顺序排查
2. JAR 包大小是判断完整性的最简单方式（Spring Boot fat jar 不可能只有几百 KB）
3. 工具不可全信——写文件后必须验证，用"验证代替假设"
4. DTO 和 API 接口必须前后端同步设计，避免参数被静默丢弃
5. 配置项名称是精确匹配的，差一个点都不行
6. 每次部署后 curl 测试关键接口，不要假设"编译成功 = 部署成功"
